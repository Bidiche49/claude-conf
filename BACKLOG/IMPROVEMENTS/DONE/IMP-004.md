# IMP-004: Supervisor — scope strict sur le commit en multi-supervisors

**Type:** Improvement
**Statut:** Fait
**Priorite:** Haute
**Complexite:** S
**Scope:** both
**Tags:** supervisor, multi-session
**Date creation:** 2026-03-20

---

## Description

Quand plusieurs supervisors tournent en parallele, chacun voit le working tree complet (git status, git diff) mais ne doit committer que les fichiers de SON scope (ses tickets). Actuellement la regle existe dans la section MULTI-SUPERVISORS mais elle est enterree en bas du prompt — les supervisors l'ignorent et tentent de committer tout ce qu'ils voient.

## Probleme concret

1. Supervisor A travaille sur FEAT-002 (pre-commit-gate)
2. Supervisor B travaille sur FEAT-003 (oneshot)
3. Les deux workers finissent et modifient des fichiers dans le working tree
4. Supervisor A fait `git status`, voit les fichiers de A ET B, et propose de tout committer
5. Resultat : Supervisor A committe le travail de B sans l'avoir valide

## Solution

### 1. Section "Validate reports" — renforcer le commit

Dans la section 5 (Validate reports), le step "commit" doit explicitement dire :

**Quand tu committe :**
- Stage UNIQUEMENT les fichiers listes dans la section "Fichiers modifies" du rapport du worker
- JAMAIS `git add .` ou `git add -A`
- Si `git status` montre des fichiers qui ne sont PAS dans le rapport → les IGNORER, ils appartiennent a un autre workflow
- Lister les fichiers stages dans le message au user avant de committer : "Je vais stager ces fichiers : [liste]. Les autres fichiers dans le working tree ne sont pas dans mon scope."

### 2. Section "Load context" — identifier le hors-scope

Au chargement de contexte (step 1), si `git status` montre des fichiers modifies, le supervisor doit :
- Les noter
- Les comparer a son scope (ses tickets)
- Si des fichiers ne correspondent a aucun de ses tickets → mentionner "X fichiers hors scope dans le working tree (probablement un autre workflow)"
- Ne JAMAIS proposer de s'en occuper

### 3. Remonter la regle — pas en bas du prompt

La regle "Only stage files from your own tickets" est dans la section MULTI-SUPERVISORS en bas du prompt. Elle doit aussi apparaitre dans le workflow de commit (section 5) car c'est LA ou le supervisor prend la decision de stager.
