## Description

5 modules n'ont aucun test automatise. Boy scout rule : chaque module avec un script shell devrait avoir au minimum des tests basiques validant que le hook/install fonctionne.

## Modules concernes

| Module | Fichiers a tester | Type de test |
|---|---|---|
| `pre-commit-gate` | `hooks/pre-commit-gate.sh`, `install.sh` | Hook detecte `git commit`, ignore `git push`. Install idempotent. |
| `backlog-kit` | `hooks/backlog-guard.sh`, `install.sh` | Hook bloque ID duplique, autorise ID libre. Install idempotent. |
| `api-contract` | `hooks/api-contract-reminder.sh`, `install.sh` | Hook detecte controller.ts, ignore utils.ts, silencieux sans contrat. Install idempotent. |
| `claude-md-kit` | `install.sh` | Install idempotent, commandes copiees. |
| `setup-project` | `install.sh` | Install idempotent, commandes copiees. |

## Approche

Pour chaque module, creer un `tests/test.sh` (ou `tests/test-<hook>.sh`) qui :
1. Prepare un environnement temporaire (mktemp -d)
2. Simule l'input JSON du hook
3. Verifie le comportement attendu (exit code, output)
4. Nettoie

Pattern a suivre : voir les tests existants dans le repo si il y en a (backlog-kit, critical-thinking).

## Criteres d'acceptation

- [ ] Chaque module a au moins 1 fichier de test
- [ ] Les hooks sont testes avec des inputs JSON simules
- [ ] Les tests sont executables avec `bash tests/test.sh`
- [ ] Tous les tests passent
